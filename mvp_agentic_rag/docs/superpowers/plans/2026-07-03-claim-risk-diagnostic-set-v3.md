# Claim-Risk Diagnostic Set Implementation Plan v3

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` recommended, or `superpowers:executing-plans`, to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

## v3 Revision Summary

This revision keeps the original goal of building a small Claim-Risk / Evidence-Risk Diagnostic Set, but further hardens the plan for direct implementation by an agentic worker. It keeps the pilot-first workflow from v2 and adds additional contracts to prevent action-label mismatch, provenance leakage, weak pilot validation, and ambiguous evaluation results:

1. Add **Task -1: Source Run Sanity Check** before trajectory field audit.
2. Split Phase 1 annotation into a **pilot batch of 40-60 records** and a later full batch of 120-200 records.
3. Add `mining_reason` and `label_provenance` fields so automatically mined candidates are auditable and leakage is explicit.
4. Add oracle-action priority rules to reduce ambiguity between `repair_missing_hop`, `refine_query`, `read_more`, and `abstain`.
5. Add `error_attribution_matrix` after diagnostic evaluation.
6. Add training gates for `bridge_as_final_recall` and `repairable_missing_hop_recall`.
7. Clarify that this plan is a **diagnostic infrastructure plan**, not a direct full-300 performance-improvement run.
8. Add an **action normalization contract** so runtime actions such as `ordered_hop_repair`, `repair_next_hop`, `read_more_chunks`, and `continue_search` map consistently into diagnostic oracle actions.
9. Add a **candidate pool quality report** before sampling, including mining-rule distribution, provenance usage, ordered-hop signal rate, and shortage-by-bucket.
10. Add a **pilot review summary** and stronger pilot go/no-go criteria before building the full annotation batch.
11. Split evaluator outputs into **all-record metrics** and **clean-claims metrics** to avoid self-evaluation when claims are generated by the same verifier being evaluated.
12. Add provenance consistency validation, action normalization tests, unmatched-ID categories, and training-decision gates split by verifier, controller/calibration, and repair/retrieval bottlenecks.

## Goal

Build a small Claim-Risk / Evidence-Risk Diagnostic Set that independently evaluates claim-level verifier and controller decisions before deciding whether to train a verifier model.

The diagnostic set should answer:

1. Is the current claim-level diagnosis accurate enough?
2. Is final performance bottlenecked by verifier diagnosis, controller action mapping, repair query quality, or retrieval ceiling?
3. Does claim-level risk diagnosis outperform a response-level critic baseline on action decisions?
4. Would a lightweight verifier or calibrator have clear expected ROI?

The first stage intentionally avoids training a verifier model.

---

## Decision

**Do not train a verifier model in this stage.**

Training becomes a later branch only if diagnostic evidence shows that verifier diagnosis is the main bottleneck.

Default next route should be:

```text
continue_without_training unless diagnostic evidence clearly shows verifier bottleneck
```

This plan should not be evaluated by whether it directly improves `answer_f1`, `coverage`, or `4-hop coverage`. It succeeds if it makes the current bottleneck measurable and gives a defensible decision about whether to train, calibrate, or instead repair controller/retrieval logic.

## Artifact and Leakage Policy

No diagnostic `test.jsonl` record may be used to tune prompts, policies, thresholds, repair heuristics, or training data.

Separate these concepts clearly:

```text
offline label construction:
  may use gold answer, gold supporting chain, and human review.

runtime decision:
  must not use gold answer, gold chain, or frozen diagnostic labels.
```

Every record must carry explicit provenance:

```json
"label_provenance": {
  "uses_gold_answer": true,
  "uses_gold_chain": false,
  "uses_model_output": true,
  "uses_human_review": false,
  "runtime_available": false
}
```

Every automatically mined candidate must also carry:

```json
"mining_reason": {
  "rule": "bridge_value_contains_candidate",
  "matched_fields": ["ordered_hop_binding.bound_bridge_values", "final_answer"],
  "confidence": "weak | medium | strong"
}
```

If claims were generated from the same verifier output being evaluated, this must be recorded in `metadata.claims_source` and `label_provenance`. Do not report such records as clean evidence of verifier superiority unless this provenance is disclosed.

## Validation requirements

`validate_record(record: dict) -> list[str]` should check:

- required top-level keys exist;
- all claim ids have `claim_support`;
- support labels are valid;
- oracle action is valid;
- risk type is valid;
- `state.round`, `state.max_rounds`, `state.budget_remaining`, and `state.allowed_actions` exist;
- every claim has `claim_id`, `text`, `role`, and `source`;
- `metadata.claims_source` exists;
- `mining_reason.rule`, `mining_reason.matched_fields`, and `mining_reason.confidence` exist;
- `label_provenance` exists and contains all required booleans;
- `oracle_action=answer` cannot have `should_abstain=true`;
- `oracle_action=repair_missing_hop` should include at least one repair target field;
- `bridge_as_final=true` implies `wrong_target=true` unless the record is marked `unclear` or `adjudication_needed`;
- `final_answer_supported=false` and `oracle_action=answer` is invalid unless the record is explicitly marked `unclear`;
- if `mining_reason.rule` uses exact match, token F1, gold mismatch, gold bridge values, or gold chain fields, then `label_provenance.uses_gold_answer` or `label_provenance.uses_gold_chain` must be true and `label_provenance.runtime_available` must be false;
- `oracle_action` not present in raw `state.allowed_actions` should produce a warning, not a hard failure, because runtime action lists may be incomplete;
- `action_metadata.diagnostic_action`, if present, must equal `normalize_runtime_action(action_metadata.runtime_action)`.

## README requirements

Include:

- benchmark purpose;
- Phase 1 scope: MuSiQue-only development diagnostic set;
- Phase 2 scope: MuSiQue + HotpotQA + 2Wiki core diagnostic set;
- explicit note that this set evaluates decision points, not only final QA;
- leakage rule: `dev.jsonl` may tune prompts/policies, `test.jsonl` is frozen;
- data artifact policy: which files are committed and which are regenerated;
- warning that this is not a replacement for the main QA benchmark.

## Annotation guidelines requirements

Include label definitions:

- `supported`: current evidence explicitly supports or minimally entails the claim;
- `unsupported`: evidence is insufficient;
- `contradicted`: evidence states an incompatible fact;
- `critical_missing_claim`: final answer cannot be safely produced without it;
- `noncritical_missing_claim`: useful for explanation but not required for final answer;
- `wrong_target`: answer is grounded but answers the wrong entity/relation/slot;
- `bridge_as_final`: answer is an intermediate entity, not the requested final answer.

## Claim-source rules

- Prefer existing trajectory claims when they are explicit and auditable.
- Add one final-answer claim when no explicit final claim exists.
- Use ordered-hop / missing-hop fields for bridge and repair-target claims.
- Mark `claims_source=manual_required` if no reliable claim can be derived automatically.
- Do not compare verifier performance on records whose claims were generated from the same verifier output unless this provenance is reported.

## Oracle-action priority rules

Use this order to reduce annotation ambiguity:

```text
1. answer
   If all critical final-answer claims are supported, final target is correct,
   and there is no unresolved final-slot conflict.

2. repair_missing_hop
   If the current candidate is wrong-target or bridge-as-final,
   and a missing hop can be identified with anchor_entity and target_relation.
   Prefer repair_missing_hop over refine_query when the anchor and relation are known.

3. disambiguate_conflict
   If evidence contains incompatible candidates for the same final slot,
   or source/time ambiguity must be resolved before answering.

4. read_more
   Only if a retrieved source is clearly relevant, the current chunk contains partial relation evidence,
   the missing answer or relation is likely in a neighboring chunk or same document,
   and reading more from that source is more appropriate than issuing a new query.

5. refine_query
   If a critical claim is missing and budget remains,
   but no reliable repair anchor/relation is available.

6. abstain
   Only if evidence is insufficient and no reliable repair target exists,
   budget is exhausted, repeated retrieval gives no new evidence,
   contradiction cannot be resolved, or candidate extraction fails after repair.
```

## Required annotation examples

The guidelines must include at least these examples so reviewers apply labels consistently:

1. **Bridge-as-final / repairable missing hop**
   - Candidate: `Nieuwe Maas River`
   - Gold final answer: `Het Scheur`
   - Label: `bridge_as_final=true`, `wrong_target=true`
   - Oracle action: `repair_missing_hop`, with anchor entity `Nieuwe Maas River` and target relation like `mouth / continuation / watercourse`.

2. **Role / relation-chain mismatch**
   - Candidate: `Salma Hayek`
   - Gold final answer: `Maria Bello`
   - Label: `wrong_target=true`; use `bridge_as_final=true` only if the candidate is an explicit intermediate value in the evidence chain.

3. **Sufficient evidence but no candidate**
   - `final_slot_covered=true`, but candidate is `UNKNOWN` or empty.
   - Label: `candidate_extraction_failure` in notes or metadata; oracle action usually `repair_missing_hop` if an extraction/anchor target exists, otherwise `read_more` or `abstain` depending on evidence.

4. **4-hop partial chain**
   - Verified prefix exists, next relation missing.
   - Label: `repairable_missing_hop`; oracle action `repair_missing_hop`, not generic `refine_query`, when anchor entity and target relation are known.

## Adjudication rules

- Every `test.jsonl` record must be human verified.
- If two reviewers disagree on `oracle_action`, keep the case in `dev.jsonl` or mark it `adjudication_needed`; do not put it in frozen test.
- If evidence is ambiguous and no external gold evidence resolves it, use `unclear` for claim support and avoid treating the example as a positive/negative for that metric.
- If contradiction candidates are too scarce in natural trajectories, record scarcity rather than fabricating a headline contradiction metric.

## Preliminary risk-type heuristics

Candidate buckets:

- `supported_answer`: final action is answer and exact/F1 is high with no final unsupported risk;
- `critical_gap`: missing critical hops or insufficiency before answer;
- `wrong_target`: answer is not exact and evidence appears to bind a bridge/final mismatch signal;
- `bridge_as_final`: answer appears in bridge values but not as gold final answer;
- `insufficient_evidence`: final action is abstain due to insufficient evidence;
- `no_new_evidence`: later retrieval round produced no new evidence;
- `repairable_missing_hop`: verified prefix exists with next missing relation or repair target;
- `contradiction`: unsupported claim status includes contradicted.

Keep these as preliminary labels with `annotation_status=auto`; human verification remains required.

## Prediction JSONL format

```json
{
  "id": "same-id-as-diagnostic-record",
  "predicted_claim_support": {"c1": "supported"},
  "predicted_evidence_sufficiency": "sufficient",
  "predicted_wrong_target": false,

  "predicted_repair_target": {
    "missing_hop": null,
    "anchor_entity": "",
    "target_relation": "",
    "suggested_query": ""
  },
  "prediction_source": "trajectory_export",
  "source_run": "..."
}
```

## Metric views

The evaluator must report two views:

```text
all_records_metrics:
  includes every validated record.

clean_claims_metrics:
  excludes records where claims were generated by the same verifier output being evaluated, unless the metric is explicitly about policy/action rather than verifier claim support.
```

Clean claim metrics should normally include records with `claims_source` such as `manual_required`, `final_answer`, `ordered_hop`, or `gold_chain`, while reporting provenance clearly.

## Metric groups

### diagnostic_metrics

- claim support accuracy;
- evidence sufficiency macro-F1;
- critical gap precision/recall/F1;
- wrong target accuracy;
- wrong target precision/recall/F1;
- bridge-as-final accuracy;
- bridge-as-final precision/recall/F1;
- contradiction precision/recall/F1 when enough examples exist;
- repairable_missing_hop precision/recall/F1.

### policy_metrics

- oracle action accuracy from predicted diagnosis/action;
- answer/refine/repair/read_more/disambiguate/abstain macro-F1;
- abstention precision/recall;
- repair target exact match;
- repair target partial match for anchor entity and target relation;
- balanced action accuracy;
- missed repair opportunity rate;
- over-abstain rate;
- unsafe answer rate.

Definitions:

```text
missed_repair_opportunity_rate =
  count(gold_action=repair_missing_hop and predicted_action in {abstain, refine_query})
  / count(gold_action=repair_missing_hop)

over_abstain_rate =
  count(gold_action in {answer, repair_missing_hop, read_more, refine_query} and predicted_action=abstain)
  / count(gold_action in {answer, repair_missing_hop, read_more, refine_query})

unsafe_answer_rate =
  count(gold_action != answer and predicted_action=answer)
  / count(predicted_action=answer)
```

### oracle_diagnosis_controller_metrics

If a controller export is available, report:

- action accuracy when gold diagnostic labels are provided to the controller;
- macro-F1 by action;
- gap between oracle-diagnosis action accuracy and current prediction-based action accuracy.

This gap is the verifier-bottleneck estimate.

If the interface is missing, report:

```json
{"oracle_diagnosis_controller_status": "not_available"}
```

Do not use unavailable metrics to justify verifier training.

# Task 9: Decision Report for Verifier Training

**Files:**

- Create after evaluations: `mvp_agentic_rag/diagnostic_sets/claim_risk_v1/results/training_decision_report.md`

## Required comparisons

Compare:

- current claim-risk verifier/controller;
- prompt verifier if exportable;
- response-level critic baseline, or `response_level_baseline_status=deferred` if not implemented in Phase 1;
- oracle diagnostic labels as upper bound;
- oracle-diagnosis controller action accuracy, or `oracle_diagnosis_controller_status=not_available` if the controller export interface is missing.

## Training and routing gates

Do not train a verifier unless the diagnostic report shows a verifier-specific bottleneck. Split the decision into three gate groups.

### A. Verifier training gate

Consider verifier training only if at least one is true:

- claim support accuracy is below `0.80` on clean-claims metrics;
- critical gap F1 is below `0.75`;
- wrong-target detection accuracy is below `0.80`;
- bridge-as-final recall is below `0.80`;
- repairable-missing-hop recall is below `0.70`;
- cross-dataset diagnostic performance drops sharply after Phase 2.

### B. Calibrator / controller gate

Prefer `train_lightweight_calibrator` or `fix_controller_mapping` if:

- diagnostic metrics are acceptable but oracle action accuracy is low;
- over-abstain rate is high;
- unsafe answer rate is low but missed repair opportunity rate is high;
- oracle-diagnosis controller action accuracy exceeds current prediction-based action accuracy by at least `10` absolute points.

### C. Repair / retrieval gate

Prefer `fix_repair_query_or_retrieval` if:

- repair target exact/partial match is good;
- repairable_missing_hop detection is good;
- but repair_closed_final remains low in downstream runtime runs;
- or retrieval evidence gain is low after correct repair targets.

LLM verifier cost can justify a smaller model or calibrator only if calls dominate runtime cost and diagnostics show the diagnosis itself is worth preserving.

## Allowed decisions

1. `continue_without_training`
   - verifier is adequate; bottleneck is repair, retrieval, or controller.
2. `fix_controller_mapping`
   - diagnosis is good, but predicted action differs from oracle action.
3. `fix_repair_query_or_retrieval`
   - repair target is correct, but retrieval/evidence closure fails.
4. `train_lightweight_calibrator`
   - diagnosis is mostly good but action/risk thresholds are poorly calibrated.
5. `train_claim_evidence_classifier`
   - claim support/contradiction is the clear bottleneck.
6. `train_generative_verifier`
   - only if enough high-quality labels exist and deployment cost is central.

Recommended default:

```text
continue_without_training unless diagnostic evidence clearly shows verifier bottleneck
```

```text
9. Task 5 pilot annotation export
10. Task 5.5 pilot review summary
```

Do not build the full 120-200 record batch until the pilot summary returns `go`.

This plan intentionally does not change runtime agent behavior. Runtime fixes such as `partial_chain_next_hop_repair`, Logic-DAG planning, repair query rewriting, or verifier training should be chosen only after the diagnostic report identifies the bottleneck.

# Non-Goals

- Do not train a verifier model in this stage.
- Do not build a 4000-label RAG error taxonomy.
- Do not replace the main QA benchmark.
- Do not tune on diagnostic test split.
- Do not claim cross-dataset generalization from MuSiQue-only diagnostic data.
- Do not claim direct full-300 performance improvement from this diagnostic plan alone.
